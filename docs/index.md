## Phase 4 + Phase 5: Modification and Extension (kafka_consumer_teja.py)

**File:** `src/streaming/kafka_consumer_teja.py`

**Scenario:** Extend the case example with two side-by-side live charts —
one showing the pre-tax sale value per message, and one showing which
regions generate the most revenue as the stream runs.

### What changed from the case example

The single chart was replaced with **two side-by-side charts** using
`plt.subplots(1, 2)`:

- **Chart 1 (left):** Line chart of **subtotal (pre-tax)** per message
  — Phase 4 change: plots `subtotal` instead of `total` to show raw sale
  value before regional tax is applied.
- **Chart 2 (right):** Horizontal bar chart of cumulative total sales by region
  — Phase 5 extension: a `region_totals` dictionary accumulates each region's
  running total and feeds the bar chart on every message.

### What the consumer does to each message

1. Validates required fields against the data contract
2. Enriches with `subtotal`, `tax_amount`, and `total` using region tax rates
3. Updates both live charts
4. Updates `region_totals[region_id]` with the new message total
5. Writes the enriched record to `data/output/consumed_sales_teja.csv`

### What the charts show

- The **subtotal chart** shows the raw sale value before tax, which can differ
  significantly across regions with different tax rates.
- The **region bar chart** reveals which regions (e.g., US-TX, CA-QC) accumulate
  the most revenue over the stream, making regional concentration immediately visible.

### Patterns and insights

With the sample data, US-TX dominates early due to higher-quantity orders.
The bar chart makes it easy to see regional concentration at a glance,
something the line chart alone cannot show.

## Custom Project

### Dataset

The producer reads from **`data/sales.csv`**, a simulated e-commerce sales
dataset with 178 records.

Each record contains the following fields:

| Field | Description |
| --- | --- |
| `order_id` | Unique UUID for each order |
| `datetime` | ISO 8601 timestamp of the order |
| `region_id` | Region code (e.g., `US-TX`, `CA-QC`) |
| `currency_code` | Currency (e.g., `USD`, `CAD`) |
| `product_id` | Product identifier (e.g., `PY-STREAM-005`) |
| `unit_price` | Price per unit |
| `quantity` | Units purchased |
| `is_online` | Whether the order was placed online |
| `customer_id` | Customer identifier |
| `is_new_customer` | Whether this is the customer's first order |
| `device_type` | Device used (tablet, mobile, desktop) |
| `payment_method` | Payment method (e.g., paypal) |
| `referral_source` | How the customer found the product |
| `discount_code` | Discount code applied (may be empty) |
| `customer_note` | Optional free-text note from the customer |

The original sales dataset was used without modification.

### Kafka Messages

The producer reads each row from `sales.csv` and sends it as a Kafka message.

- **Topic:** `streaming-04-visualization-case`
- **Message key:** `region_id` — each message is keyed by the sales region,
  which means messages from the same region always go to the same partition.
- **Message fields:** all 15 fields from the CSV, sent as-is with no changes.
  The consumer adds internal metadata fields (`_kafka_key`, `_kafka_partition`,
  `_kafka_offset`) when the message is received.

### Consumer Processing

The consumer (`kafka_consumer_teja.py`) receives each Kafka message and runs
the full processing pipeline:

- **Receives:** raw sales records as dictionaries from the Kafka topic
- **Message count:** up to 1000 messages per run; stops on a 10-second timeout
  with no new messages
- **Validates:** checks that all required fields are present using the data contract;
  rejects and skips messages that fail validation
- **Enriches:** computes three derived fields not present in the original CSV:
  - `subtotal` = `unit_price` × `quantity`
  - `tax_amount` = `subtotal` × regional tax rate (looked up from `data/regions.csv`)
  - `total` = `subtotal` + `tax_amount`
- **Logs:** per message — subtotal, tax, total, running total, accept/reject status,
  order ID, region, consumed count, and running statistics (total, average, min, max)
- **Writes:** enriched records to `data/output/consumed_sales_teja.csv`
- **Charts:** updates two live Matplotlib charts as each message arrives

### Experiments

**Phase 4 — Technical Modification:**
Changed the live chart to plot `subtotal` (pre-tax amount) instead of `total`.
This required overriding `update_live_chart` from the case module with a local
version that reads `message["subtotal"]` instead of `message["total"]`.
Chart title and y-axis label were updated to reflect the change.

**Phase 5 — Apply the Skills:**
Added a second live chart side-by-side using `plt.subplots(1, 2)`:
a horizontal bar chart that accumulates total sales per region in a
`region_totals` dictionary and redraws on every message.
Both the local chart helpers (`init_live_charts`, `update_live_charts`) and
all downstream function signatures (`initialize_output`, `process_message`,
`consume_messages`) were updated to carry the extra chart state.
Both phases were then merged into a single file (`kafka_consumer_teja.py`).

### Results

When the producer and consumer were run together:

- The producer published all 178 messages from `sales.csv` to the Kafka topic,
  keyed by `region_id`, at a 2-second interval.
- The consumer read from the beginning of the topic, validated and enriched each
  message, and updated both live charts in real time.
- Chart 1 showed each sale's pre-tax subtotal fluctuating message-by-message
  as different products and quantities appeared.
- Chart 2 built up a regional picture — US-TX accumulated the highest total
  early due to higher-quantity multi-unit orders, while CA-QC appeared with
  smaller but consistent contributions.
- The final summary logged total sales, average sale, min, and max across all
  consumed messages.

### Interpretation

**What changed from the original example:**
The case consumer displayed a single line chart of `total` per message.
The modified consumer (`kafka_consumer_teja.py`) shows `subtotal` (pre-tax)
in Chart 1, making it possible to see the raw sale value before regional tax
is applied — which varies from 8.25% (US-TX) to 14.975% (CA-QC).
Chart 2 adds a completely new dimension: cumulative revenue by region,
which the original example did not visualize at all.

**What the stream revealed:**
Each message arrived as an independent event, but the bar chart made it
possible to see a pattern emerge across events — regional sales concentration
became visible within just a few messages.
High-quantity orders in US-TX inflated that region's bar quickly,
while lower-quantity single-item orders in other regions grew more slowly.

**What this could tell a business:**
A business running this consumer in production could monitor in real time
which regions are driving revenue, whether new orders are high or low value,
and how pre-tax revenue compares to post-tax collected amounts across regions
with different tax obligations.

**Business intelligence gained:**
Streaming visualization turns a sequence of individual transaction events
into an ongoing regional revenue dashboard — no batch job or database query
required. Decisions about regional promotions, pricing adjustments, or
inventory allocation could be informed by what the stream shows as it runs.
